# Agentic Harness Token Optimization Guide

## 1. Executive Summary
The Hub-and-Spoke Agentic Harness optimizes token ingestion through extreme role isolation and aggressive indexer usage. Compared to traditional Monolithic agents that ingest the entire system prompt and accumulate vast conversation history, our architecture achieves **~30% token savings within just 4 interactions**, scaling to **60-80% savings** over long, multi-task development sessions.

## 2. The Monolith Problem
Traditional LLM agents use a "Monolith" approach:
- They load **all rules** (planning, implementing, reviewing, testing) into a single, massive system prompt.
- They append the **full conversation history** (including massive code block outputs and file reads) to every subsequent turn.
- By task 4 or 5, a single turn can cost 20,000+ tokens just to ask a simple question, leading to massive API costs and degraded reasoning performance.

## 3. The Hub-and-Spoke Solution
The Agentic Harness solves this via delegation and context isolation:
1. **The Orchestrator**: Acts as a lightweight router (base cost of ~4,500 tokens). It only remembers high-level summaries of completed tasks. It **NEVER** stores the raw code generated by subagents in its history.
2. **Specialized Subagents (Architect, Implementer, etc.)**: These agents are spun up "fresh" for each specific task. They only load their specific instructions (~1,500 tokens). Once they complete their task, their context is destroyed.
3. **Indexer MCP**: Instead of reading full files, all agents are mandated to use `indxr` tools (`summarize`, `get_callers`, `explain_symbol`), meaning codebase discovery costs hundreds of tokens instead of thousands.

## 4. The Adversarial Reality Check (The "Routing Tax")
An adversarial audit of this architecture reveals a critical trade-off: **The Hub-and-Spoke model is mathematically LESS efficient for the very first task in a session.**
- Because the Orchestrator cannot perform work directly, fixing a single typo requires paying the ~4,500 token "Routing Tax" for the Orchestrator, PLUS the tokens to spin up the Implementer. 
- A Monolith would just fix the typo immediately. 
- Furthermore, the `indxr` MCP does not make codebase discovery "free"; it compresses it. Reading a 5,000-token file is replaced by reading a 2,000-token indexer summary. 
- **The True Value:** The "massive efficiency" is not derived from eliminating initial costs; it is derived from **preventing history bloat**. The Harness pays a higher upfront tax to guarantee the context window stays completely flat (at ~9,700 tokens per cycle) indefinitely, while the Monolith's cost compounds exponentially until it collapses.

## 5. How to Balance Monolith vs. Harness
To mitigate the "Routing Tax" for simple tasks, the Orchestrator uses a **Complexity Assessment Hierarchy**:
*   **Tier 0 (Zero Delegation Path)**: *Best of the Monolith.* For simple Q&A, stack trace explanations, or conceptual requests requiring NO file modifications and NO deep research, the Orchestrator acts monolithically and answers directly. Zero routing tax.
*   **Tier 1 (Fast Path)**: *Hybrid.* For single-file edits or typos, the Orchestrator bypasses the Planning and Architecture phases entirely and delegates straight to the Implementer.
*   **Tier 2 (Standard Path)**: *Full Harness.* For multi-file features or complex bugs, the Orchestrator enforces the full workflow (Brainstorm -> Plan -> Implement -> Verify), absorbing the higher initial token cost to guarantee long-term stability and architectural safety.

## 6. Context Caching (Cross-Session)
Because the Agentic Harness stores its rules, agent definitions, and system prompts in static Markdown files (`orchestrator.md`, `core_mandates.md`), it is perfectly designed to exploit **Explicit Context Caching**.
- **Standard Sessions**: Normally, restarting a session means paying to re-ingest all rules from scratch.
- **Explicit Context Caching**: The underlying engine uploads the static rules to the AI provider's Cache once. The provider returns a `Cache ID`.
- **Cross-Session Persistence**: If you open a new terminal window or start a new task tomorrow, the engine just passes the `Cache ID`. As long as the files haven't changed, this results in **near-zero latency** and drastically reduced costs for the foundational system prompts, across *any* session using that API key.

## 7. Model Tiering & Cognitive Load (Unlocking 'Flash' Models)
Beyond just saving tokens, the Hub-and-Spoke model structurally improves the reasoning capabilities of smaller, faster models (like Gemini Flash). 

When a Monolithic agent is given a 20,000-token prompt containing instructions for planning, TDD, linting, architecture, and previous conversation history, smaller models suffer from the **"lost in the middle" effect**—they forget constraints or hallucinate because their attention mechanism is overwhelmed.

By isolating roles, the Agentic Harness acts as a cognitive filter:
- **Hyper-Focused Prompts**: When the `implementer` is spun up, it is *only* given the rules for writing code and the exact plan. It doesn't need to know how to route tasks or write design docs.
- **Smaller Context = Higher Accuracy**: Because the context window is kept small and relevant (~1,500 - 3,000 tokens), a Flash model can maintain near-perfect instruction following.
- **Tiered Routing**: The Orchestrator's `<model_selection_policy>` explicitly dictates using `pro` models for deep reasoning (Planning, Architecture) and `flash` models for execution (Implementing, Linting). This hybrid approach achieves "Pro-level" quality at "Flash-level" speeds and costs.
